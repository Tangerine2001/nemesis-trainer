# Monetization And Risk Notes

The business goal is a useful tool that can passively earn ad revenue without requiring expensive infrastructure.

## Revenue Model

Start with ads on content and result pages:

- matchup explainers
- boss challenge pages
- counter guides
- archetype pages
- daily challenge archives

Avoid placing ads inside critical controls or making the main tool hard to use.

Later optional revenue:

- ad-free supporter mode
- saved teams
- private notes
- exports
- advanced batch analysis

Do not add payment or account systems until the free loop has evidence of repeat use.

## Backend Cost Control

Avoid in the MVP:

- persistent battle servers
- live matchmaking
- realtime spectator features
- chat
- ranked ladders
- high-volume LLM generation

Prefer:

- static hosting
- cached JSON
- deterministic generation
- client-side analysis where practical
- one small stateless simulation worker when needed

## Trademark And Content Risk

Avoid public branding that includes:

- `Pokemon`
- `Cynthia`
- `GPT`
- official game titles as the primary product name

Use original copy, original UI, and carefully licensed assets. Do not assume official sprites, names, logos, or artwork are safe for an ad-supported site.

## Advertising Quality

Ad-supported pages need real utility.

Each generated or static page should offer original analysis:

- why the matchup is hard
- what sequence creates pressure
- what the user can change
- how an archetype functions
- what assumptions the analysis uses

Thin pages that only mirror usage stats or scraped text are risky for search quality and ad approval.
